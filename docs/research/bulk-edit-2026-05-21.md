# Bulk-edit (PR1 selection + delete) — edge-case research 2026-05-21

Component: opt-in bulk multi-select + bulk-DELETE on the shared `ListPage`
(covers all 8 taxonomy tables). Frontend orchestration mirroring PR #141 audit
bulk-revert. QG Step 2 evidence.

## Edge cases covered

1. **Empty selection**: toolbar hidden at 0 selected (`bulkEnabled && selectedCount > 0`);
   `handleBulkDelete` early-returns when `selectedItems.length === 0`. No accidental
   no-target delete.
2. **All-fail batch**: every mutation rejects -> error-variant toast
   `bulk.deleteAllFailed`. Selection is cleared and the list refetched, so the
   still-present rows reflect authoritative server state (no phantom removal).
3. **Mixed success/fail**: warning-variant toast `bulk.deletePartial {success, failed}`.
   Verified with a 2-ok / 1-fail mock against `Promise.allSettled`: tally counts
   `fulfilled` vs `rejected`; ordering preserved because `selectedItems` keeps
   `items` order (test `reports partial-failure counts when some deletes reject`).
4. **Stale selection on navigation**: selecting a row then changing the search
   query (or page/pageSize/sort) clears the selection via a `useEffect` keyed on
   `criteria`. Prevents applying a delete to rows no longer visible — the PR #141
   Codex P2 `stale-selection-leak` lesson (test `clears selection when the search
   query changes`).
5. **>5 selected**: confirm dialog lists the first 5 `getItemLabel` values plus a
   `bulk.andMore {count}` overflow line, avoiding an unbounded dialog.

## Authoritative-state (no optimistic mutation) rationale

Per the spec's approved decision, there is no optimistic cache mutation. With
partial-failure the post-operation `refreshList()` (query invalidate -> refetch)
is the source of truth: succeeded rows vanish, failed rows reappear. "Rollback"
is therefore implicit, removing the risk of reconciling a half-applied batch.

## Tuning note

An N-row bulk delete issues N parallel single-DELETE requests via
`Promise.allSettled`, bounded by the visible page size (<=50). Taxonomy seed
datasets are small (<=53-species fixture), so no request batching/throttling is
needed at current scale. Revisit (chunked dispatch or a backend batch endpoint)
only if a single page exceeds ~500 rows.
